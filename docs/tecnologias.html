Tecnologias · DataSUS-RS

Mapeamento das tecnologias usadas em cada parte do pipeline — da ingestão batch ao consumo em tempo real — com justificativa de cada escolha.

UFRGS · NF01006 — Projeto de Banco de Dados · 2026

🌿 Ingestão Backfill + Cronjob mensal

Coleta dos dados das fontes (DataSUS, IBGE, Atlas Brasil, ObservaPOA, SMAMUS) pra dentro do data lake.

TecnologiaOnde usoPor que escolhi
Python 3.11 Linguagem dos scripts de ingestão Único runtime que atravessa todo o pipeline; bibliotecas científicas de primeira
pysus (Fiocruz) Download SIH e CNES do FTP DataSUS Biblioteca oficial mantida pela Fiocruz; abstrai conversão .dbc → .parquet e cache local
urllib + requests (stdlib) API IBGE SIDRA (Censo 2022), Atlas Brasil 2010, ObservaPOA, SMAMUS ArcGIS REST APIs públicas RESTful retornam JSON — não preciso de cliente especializado
supercronic Cronjob mensal de ingestão dentro do container Versão de cron amigável a containers (sem precisar de root/systemd)

🟫 Camada Bronze Dados brutos do DATASUS

Preserva os dados exatamente como vieram do pysus, sem alterações — auditabilidade total.

TecnologiaOnde usoPor que escolhi
Parquet Formato canônico dos arquivos brutos Colunar + compressão zstd (~5×) + schema embarcado + pushdown de filtros
PyArrow Escrita e leitura dos arquivos parquet Padrão de mercado Apache Arrow; integração nativa com DuckDB e Pandas
Hive partitioning uf=/ano=/mes= Particionamento físico dos arquivos DuckDB faz pushdown de filtros temporais automático — não preciso indexar

🥈 Camada Silver Limpo + tipado + padronizado

Aplica a lógica de limpeza UMA vez (derivações cid3, csap_flag, idade_anos) — todas as tabelas gold consumem daqui.

TecnologiaOnde usoPor que escolhi
DuckDB Transformações bronze → silver via COPY ... TO ... 5-10× mais rápido que Spark single-machine; SQL completo; zero overhead JVM
SQL (DuckDB dialect) Lógica de limpeza, casts, derivações (cid3, csap_flag, idade_anos) Declarativo, fácil de revisar; padrão universal de transformação de dados
scikit-learn train_test_split estratificado do dataset ML (silver-grain) Padrão da indústria pra split com random_state; estratificação por target preserva prevalência

🥇 Camada Gold Tabelas dashboards + modelo

Agregações pré-computadas (sub-segundo no dashboard) + dataset ML versionado + modelo treinado.

TecnologiaOnde usoPor que escolhi
Parquet particionado por ano 5 tabelas gold (continuum_cid, hospital_panel, leitos_municipio, mortalidade, ml_dataset) Mesma razão do bronze; agregações pré-computadas → leitura sub-segundo
DuckDB Agregações silver → gold (GROUP BY) e queries dos dashboards Aproveita o mesmo motor da camada silver; cache de filesystem
joblib / pickle (futuro) Serialização do modelo treinado (modelo.pkl) Convenção scikit-learn / LightGBM; preserva pesos + preprocessamento

📊 Consumo — Dashboards UF · Município · Hospital

13 páginas Streamlit consultando direto os parquets gold via DuckDB embarcado — sem servidor de banco.

TecnologiaOnde usoPor que escolhi
Streamlit Multipage app com 13 dashboards (uma por nível de granularidade) Python-native (mesmo stack do pipeline); hot-reload; multipage nativo; sem precisar React/JS separado
DuckDB embed Cada página Streamlit roda DuckDB embarcado pra consultar gold Não precisa de servidor de banco; query direta sobre parquet
Plotly Gráficos interativos (line, bar, funnel plot, sankey) Interatividade rica + hover + zoom; integração nativa Streamlit
Folium + streamlit-folium Mapa coroplético do Silêncio Epidemiológico Leaflet em Python; melhor pra coropléticos detalhados (94 bairros POA)
GeoJSON 2 shapefiles (RS 497 municípios + POA 94 bairros) Padrão web; renderiza nativamente em Plotly e Folium
CSS custom inline Estilo do pergunta_box, callouts, status colors Personalização sem framework adicional
@st.cache_data / @st.cache_resource Cache de queries DuckDB e conexões Sub-segundo nas re-renderizações da UI

🚨 Consumo — Paciente Streaming + Modelo + Form

F6 motor de decisão: evento de triagem → Kafka → consumer com modelo carregado → score + persistência Postgres.

TecnologiaOnde usoPor que escolhi
Apache Kafka Tópicos admissoes-hospitalares (entrada) e scores-risco (saída) Decoupling producer-consumer; replay capability; padrão MLOps em produção
kafka-python Cliente Producer/Consumer em Python Maduro; integra com asyncio se preciso
Zookeeper Coordenação do cluster Kafka Dependência obrigatória do Kafka 7.6
PostgreSQL 16 Armazém de alertas / predições (audit log) ACID pra eventos críticos; índice B-tree por timestamp; JSONB pra payload semi-estruturado
psycopg2-binary Driver Postgres em Python Cliente oficial; binary build elimina compilação
Streamlit form Tela de inserção de dados do paciente + visualização do score Mesma stack do dashboard analítico → coesão visual
scikit-learn / LightGBM (futuro) Carregamento do modelo.pkl em memória para inferência Padrão da literatura; predict_proba direto

🛠️ Cross-cutting Orquestração + DevOps

Infraestrutura compartilhada entre todas as camadas — garante reprodutibilidade e versionamento.

TecnologiaOnde usoPor que escolhi
Docker + Docker Compose Orquestra 8 serviços (Postgres, Zookeeper, Kafka, Kafka UI, Streamlit, Jupyter, ingestor, Spark) Reprodutibilidade absoluta; setup zero-config; isolamento de runtime
Bash scripts Wrappers de execução (docker compose run --rm ...) Padrão Unix; legível por qualquer dev
Git + GitHub Versionamento de código + dados de referência leves Indispensável; permite revisão histórica
Conventional Commits Convenção de mensagens (feat:, fix:, docs:) Histórico legível; permite changelog automatizado
