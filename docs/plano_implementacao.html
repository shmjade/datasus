Escopo do Trabalho

Objetivos

Construir uma plataforma de dados capaz de correlacionar a latência na regulação assistencial com desfechos de mortalidade, provendo uma camada de decisão em tempo real para otimização do fluxo de pacientes críticos.

Pipeline Batch (ETL Histórico)

Ingestão e transformação de dados históricos do SIH para calcular a "Janela de Risco" — tempo entre solicitação de internação e desfecho adverso (óbito / agravamento).

Modelagem Relacional

Banco de dados com padrão Medallion (raw → trusted → refined) suportando consultas complexas de rastreabilidade do paciente via ID cifrado (hash SHA-256 do n_AIH).

Motor de Decisão em Tempo Real

Pipeline Kafka: ao detectar triagem de risco alto ("Vermelho"), consulta disponibilidade de leitos em unidades próximas e sugere a melhor alocação — simulado via producer Python.

Análise Preditiva e Espacial

Mapeamento de áreas de "Silêncio Epidemiológico": regiões onde alta mortalidade coexiste com baixa oferta de consultas especializadas e leitos SUS.

Integração de Dados

Fontes de Dados

Três fontes públicas federais integradas por código IBGE de município (6 dígitos), cobrindo o período 2022–2024 para a Região Sul.

Internações hospitalares
SIH/DataSUS		 SIH RDRS2401.dbc … RDRS2412.dbc ~500 k registros/ano MUNIC_MOV (6 dígitos)
Estabelecimentos de saúde
CNES/DataSUS		 CNES STRS2412.dbc, LTRS2412.dbc ~12 k estabelecimentos CODUFMUN (6 dígitos)
Indicadores socioeconômicos
IBGE / Censo 2022		 IBGE CSV por setor censitário ~47 k setores (Região Sul) CD_GEOCODM[0:6]

⚠️ Atenção — Divergência de chave IBGE: O SIH e o CNES usam código IBGE de 6 dígitos, enquanto o IBGE publica 7 dígitos (com dígito verificador). O join deve usar SUBSTRING(ibge.CD_GEOCODM, 1, 6).

Modelagem Relacional

Modelo de Dados — Padrão Medallion

Três schemas no mesmo PostgreSQL 16, refletindo as camadas de qualidade de dados. A anonimização via hash SHA-256 garante rastreabilidade sem expor dado sensível (LGPD — Art. 11).

raw — Dados Brutos
  • 📋 sih_internacoes
  • n_aih, dt_inter, dt_saida, morte…
  • 🏥 cnes_estabelecimentos
  • cnes, tp_unidade, lat, lon…
  • 🛏️ cnes_leitos
  • qt_exist, qt_sus, tp_leito…
  • 📊 ibge_indicadores
  • renda_media, idhm, pop_vulneravel…
trusted — Dados Limpos
  • 🔒 internacoes
  • n_aih_hash (SHA-256), dias_permanencia (computed), obito…
  • 🏛️ estabelecimentos
  • cnes (UNIQUE), total_leitos_sus, lat, lon…
  • FK: raw_id → raw.sih_internacoes
refined — Analítico
  • 📈 mortalidade_por_municipio
  • taxa_mortalidade (computed), PK: (municipio, competencia)
  • 🛏️ disponibilidade_leitos
  • leitos_por_1000hab (computed)
  • 🚨 alertas_triagem
  • [a criar] evento_id, cnes_origem, cnes_destino, protocolo…
  • 🗺️ silencio_epidemiologico
  • [a criar] view materializada: mortalidade ↑ × leitos ↓

Coluna computada STORED: PostgreSQL calcula dias_permanencia, taxa_mortalidade e leitos_por_1000hab automaticamente no INSERT/UPDATE — não há risco de inconsistência entre dado bruto e derivado.

2
Próxima

Ingestão de Dados (Download + Parquet)

Scripts Python que baixam os arquivos .dbc do FTP do DataSUS usando pysus, convertem para Parquet e salvam em data/raw/ particionado por UF e competência.

  • pipelines/batch/ingest/download_sih.py — baixa RD{UF}{AAAAMM}.dbc para PR, SC, RS (2022–2024)
  • pipelines/batch/ingest/download_cnes.py — baixa ST (estabelecimentos) e LT (leitos)
  • pipelines/batch/ingest/download_ibge.py — baixa CSV setorial Censo 2022 (Região Sul)
  • Converter .dbc → .parquet usando pysus.SIH / pysus.CNES
  • Particionamento: data/raw/sih/uf=RS/competencia=202401/part-*.parquet
  • Criar Makefile com target make ingest para rodar os três scripts
pysus requests pyarrow pandas
3
Pipeline Batch

ETL Batch — PySpark (raw → trusted → refined)

Jobs PySpark que lêem os Parquets da fase anterior, aplicam limpeza de qualidade, anonimização e carregam o PostgreSQL nas três camadas.

  • pipelines/batch/transform/etl_sih.py — filtros de qualidade SIH (datas inválidas, sexo=0/9, morte sem saída), hash SHA-256 do n_AIH, faixa etária, carga em raw.* e trusted.internacoes
  • pipelines/batch/transform/etl_cnes.py — join ST+LT, consolidação de leitos SUS por CNES, carga em raw.* e trusted.estabelecimentos
  • pipelines/batch/transform/etl_ibge.py — truncagem do código para 6 dígitos, carga em raw.ibge_indicadores
  • pipelines/batch/transform/aggregate.py — popula refined.mortalidade_por_municipio e refined.disponibilidade_leitos
  • Testes unitários em tests/test_etl_sih.py com pytest + PySpark local
PySpark psycopg2 hashlib pytest
4
Stream

Pipeline de Tempo Real — Kafka (Simulação de Triagem)

Produtor Python simula chegada de pacientes em pronto-socorro usando distribuição estatística derivada do SIH. Consumidor classifica o risco e consulta disponibilidade de leitos em tempo real.

  • Criar tópico triagem_eventos no Kafka (via infra/kafka/topics.sh)
  • pipelines/stream/producer.py — gera eventos JSON com: paciente_id, cnes_origem, cid_suspeito, timestamp, sinais_vitais
  • Distribuição horária de pico baseada nos dados DT_INTER do SIH
  • pipelines/stream/consumer.py — lê o tópico, aplica Protocolo Manchester (vermelho/laranja/amarelo/verde/azul), consulta refined.disponibilidade_leitos para unidades próximas
  • Grava alertas em nova tabela refined.alertas_triagem (DDL a adicionar no init.sql)
  • Criar tópico triagem_alertas para resposta ao producer

Alinhamento Acadêmico

Justificativa — NF01006

O projeto demonstra na prática os pilares da disciplina Projeto de Banco de Dados, com aplicação real sobre dados públicos de saúde.

🗄️ Modelagem Relacional

DDL completo com constraints, tipos apropriados, colunas computadas STORED, índices analíticos e comentários. Normalização adequada sem over-engineering.

🔗 Integração de Dados Heterogêneos

Três fontes com encodings, formatos (.dbc, CSV) e chaves diferentes integradas via código IBGE 6 dígitos. Desafio real de data wrangling com PySpark.

⚡ Processamento Distribuído

Spark em modo standalone com Master + Worker em Docker. Conceitos de paralelismo, particionamento de dados e execução de planos físicos aplicados a volume real (~1.5M registros).

🌊 Arquitetura Híbrida Batch/Stream

Kafka demonstra o paradigma de processamento de eventos em contraste com o ETL batch. A separação entre pipeline histórico e decisão em tempo real é um padrão central em sistemas modernos de dados.

🔐 Privacidade e LGPD

Anonimização com hash SHA-256 do n_AIH (quasi-identificador). Dado de saúde é dado sensível (Art. 11 LGPD). Rastreabilidade preservada sem exposição de PII.

📊 Inteligência Acionável

O conceito de "Silêncio Epidemiológico" transforma registros fragmentados em decisão de política pública: onde investir em novos leitos SUS para reduzir mortalidade evitável.

Nota sobre Cooperação: Este trabalho destina-se exclusivamente à disciplina NF01006 — Projeto de Banco de Dados e à obtenção de horas de extensão. Não há cooperação com TCC, bolsas de pesquisa ou outras disciplinas do INF.
